# Data Converter for Aggy

ユーザーが既に持っているアンケートのレイアウト情報とローデータを、Aggy形式にパースするスキル。

入力形式は様々（Excel、テキスト、CSV等）で、レイアウトとローデータが1つのファイル内にある場合も別ファイルの場合もある。このスキルの役割は、入力形式に関わらず、Aggy形式の**ローデータCSV**と**レイアウトJSON**を出力すること。

## Aggyの出力フォーマット

まず以下の2ファイルを読み、Aggyが受け付けるデータ形式を正確に理解すること:

1. **`references/data-preparation-guide.md`** — ローデータCSVとレイアウトJSONの仕様。列タイプ別の定義、無回答/回答対象外の扱い、CSVの例を含む
2. **`references/layout.schema.json`** — レイアウトJSONのJSON Schema。型ごとの必須/禁止フィールドが厳密に定義されている

この2ファイルが正（single source of truth）。以下はパース作業のための要点のみ記す。

### ローデータCSV
- UTF-8、1行目ヘッダー、2行目以降データ
- SA列: 1列に選択肢コード（数値）
- MA列: `{key}_{code}` 形式の列名、値は `1`/`0`
- WEIGHT列: 小数値
- DATE列: `YYYY-MM-DD` 形式
- 回答対象外: 空セル（NULL）
- **集計対象外の列は除外する**

### レイアウトJSON
- JSON配列。各エントリの構造は `layout.schema.json` に従う
- **`code` は必ず文字列**（`"1"` であって `1` ではない）— 最もよくあるミス

## ワークフロー

### Step 1: ファイルを読み取り、構造を把握する

ユーザーが提供したファイルを開き、**レイアウト情報**と**ローデータ**がどこにあるか特定する。

- Excelなら各シートの中身を確認する
- テキスト/CSVなら文字コード（UTF-8, Shift-JIS等）を検出して読み取る
- ファイルが1つなら、レイアウトとローデータが別シートや別セクションにないか確認する
- ファイルが複数なら、どれがレイアウトでどれがローデータか判断する

### Step 2: レイアウト情報をパースする

`layout.schema.json` を読み取り、出力すべきJSONの構造を把握した上で、入力のレイアウト情報から各設問の **key**, **type**, **label**, **items** を抽出する。

#### 集計対象の判定

`layout.schema.json` に定義された型（SA, MA, WEIGHT, DATE）に該当する設問のみを変換対象とする。以下は**集計対象外**としてスキップする:

- **自由回答**: テキスト入力の設問。カテゴリカルな集計ができない
- **数値入力**: 連続値の設問
- **システム管理列**: 回答者ID、開始/終了日時、回答時間、ステータス、クォータなど
- **その他**: 判断に迷う場合はユーザーに確認する

#### SA vs MA の見分け方

レイアウト情報に型の記載がある場合はそれに従う。ない場合の判断基準:
- **SA**: 1つのkey（列名）に対して、複数の選択肢コードが定義されている。ローデータでは1列に1つのコードが入る
- **MA**: 1つの設問に対して、選択肢ごとに個別のkey（列名）が割り当てられている。ローデータでは選択肢数分の列があり、各列の値は0/1

#### マトリクス設問の扱い

マトリクス設問（同じ選択肢を複数の対象について繰り返す形式）は、各子設問を**独立した設問**としてパースする。例: 「商品A/B/Cの満足度」→ SA x3。

### Step 3: ローデータをパースする

元のローデータを読み取り、Aggy形式のCSVに変換する:

- **文字コード**: UTF-8に変換
- **区切り文字**: カンマ区切りCSVに統一
- **集計対象外の列を除外**
- **無回答と回答対象外の変換**（後述）
- **MA列名の統一**: ローデータ上のMA列名がAggy形式（`{key}_{code}`）でない場合はリネーム
- **MA列の値**: `0`/`1` であることを確認

#### 無回答と回答対象外の扱い

Aggyでは「無回答」と「回答対象外」を明確に区別する（詳細は `data-preparation-guide.md` の該当セクション参照）:

- **無回答**（設問は表示されたが未回答）→ 選択肢コード（例: `99`）としてnに含める
- **回答対象外**（分岐条件等で設問自体が非表示）→ 空セル（NULL）としてnから除外

元データ側では、この区別の有無や表現方法がデータごとに異なる。そのため**自動判定せず、必ずユーザーに確認する**:

1. **元データに無回答と回答対象外の区別があるか？**
   - 区別がない場合 → すべて空セル（NULL=回答対象外）として扱うか、ユーザーの指示に従う
   - 区別がある場合 → 次の質問へ
2. **元データ上でそれぞれどの値で表現されているか？**（例: 無回答=`-1`、回答対象外=空白 / 無回答=`99`、回答対象外=`.` 等）
3. **無回答を集計に含めたいか？** 含める場合、レイアウトJSONの `items` に無回答用の選択肢（例: `{"code": "99", "label": "無回答"}`）を追加する

### Step 4: バリデーション

同梱の `scripts/validate_layout.py` を実行して、レイアウトJSONとCSVを検証する。

```bash
python <skill-dir>/scripts/validate_layout.py <layout.json> --csv <rawdata.csv>
```

このスクリプトは `layout.schema.json` のルールに基づき以下を検証する:
- 型ごとの必須/禁止フィールド
- `code` が文字列であること
- CSVヘッダーとレイアウト定義の整合性（列の過不足）

エラーがあれば修正して再実行する。

### Step 5: 出力

完成ファイルを保存し、ユーザーに報告:
- CSVファイルのパスと行数
- レイアウトJSONファイルのパスと設問数（SA x個, MA x個 等の内訳）
- バリデーション結果
- スキップした列の一覧と理由
- 変換時に行った判断の概要

## ユーザーとのやりとりのポイント

- レイアウト情報とローデータの**両方**が必要。片方しか提供されていない場合は、もう片方も求める
- パース処理は自動で行い、**判断に迷う点だけ**ユーザーに確認する
- 集計に含めたい設問をユーザーが限定したい場合もある
- 元データを上書きしない。常に新しいファイルとして出力する
